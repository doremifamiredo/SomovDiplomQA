# Test Plan
# План по проверке и автоматизации тестирования мобильного приложения "В Хосписе"

## 1. Цели и область тестирования

**Цель тестирования** — проверка корректной работы основных пользовательских сценариев
и снижение количества дефектов за счёт ручного и автоматизированного тестирования.

**Область тестирования:**
- Платформа: Android
- Версия ОС: Android 11
- Тестирование выполняется на эмуляторе
- Основной тип пользователя: авторизованный

---

## 2. Описание тестируемой системы

Мобильное приложение **«В Хосписе»**, разработанное для платформы Android.
Приложение написано на языке **Kotlin**.
Автоматизированные UI-тесты реализованы на языке **Java** с использованием фреймворка **Espresso**.

---

## 3. Тестовые границы (Scope)

### 3.1 Страница авторизации
- Поле ввода логина
- Поле ввода пароля
- Кнопка «Войти»
- Позитивные и негативные сценарии авторизации

### 3.2 Главная страница
- Навигационное бургер-меню
- Просмотр текущих новостей (включая ситуацию отсутствия новостей)
- Вкладка «Все новости»
- Иконка «Бабочка» (переход на страницу «Главное — жить любя»)
- Иконка «Человечек» (выход из приложения)

### 3.3 Страница новостей
- Просмотр списка новостей
- Разворачивание и сворачивание новостей
- Сортировка новостей
- Фильтрация
- Раздел «Панель управления»

### 3.4 Страница «О приложении»
- Ссылка «Политика конфиденциальности»
- Ссылка «Условия использования»
- Отображение версии приложения
- Кнопка «Назад»

---

## 4. Виды тестирования

### 4.1 Функциональное тестирование
- Ручное исследовательское тестирование (метод чёрного ящика)
- Автоматизированное тестирование:
  - позитивные сценарии
  - негативные сценарии
  - регрессионные проверки

### 4.2 UI / UX тестирование
- Проверка корректности отображения UI-элементов
- Проверка удобства навигации
- Поиск UI-ошибок  
_(выполняется вручную)_

---

## 5. Этапы выполнения работ

1. Исследовательское ручное тестирование приложения
2. Составление чек-листа
3. Написание тест-кейсов
4. Разработка автоматизированных UI-тестов
5. Прогон автотестов
6. Формирование отчёта о результатах тестирования (Allure)

---

## 6. Стратегия автоматизации

### 6.1 Объекты автоматизации
- Авторизация пользователя
- Основные пользовательские сценарии
- Критический регресс функциональности

### 6.2 Необъекты автоматизации
- Часто изменяющиеся UI-элементы
- Одноразовые проверки
- UX-оценка и субъективное удобство использования

---

## 7. Инструменты тестирования

- **Android Studio** — запуск приложения и разработка автотестов
- **Espresso** — UI-автотестирование
- **JUnit 4** — организация и управление тестами
- **Allure** — формирование отчётов о результатах прогона тестов
- **GitHub** — система контроля версий
- **Gradle** — сборка проекта и управление зависимостями
- **ADB** — взаимодействие с устройствами и эмуляторами

---

## 8. Тестовое окружение

### Устройства
- Эмулятор Android Virtual Device: Pixel 3, API 30

### Рабочая среда
- Операционная система: Windows 11
- Оперативная память: 16 ГБ
- Поддержка аппаратной виртуализации

---

## 9. Риски автоматизации

- Отсутствие формализованного технического задания
- Изменение требований в процессе тестирования
- Сложности настройки проекта автоматизации
- Необходимость поддержки автотестов при изменениях в приложении

---

## 10. Метрики качества

- Количество автоматизированных тестов
- Процент успешно пройденных тестов
- Количество дефектов, выявленных при регрессионном тестировании
- Время прогона автотестов

---

## 11. Оценка трудозатрат

- Исследовательское ручное тестирование — **8–12 часов**
- Составление чек-листа — **3–4 часа**
- Написание тест-кейсов — **6–8 часов**

### Автоматизация:
- Настройка проекта и тестового окружения — **16–24 часа**
- Реализация автотестов для основных сценариев — **60–80 часов**
- Отладка, стабилизация и прогоны — **16–24 часа**

**Итого на автоматизацию:** ~90–120 часов  
**Общие затраты:** ~18–22 рабочих дня (при 8-часовом рабочем дне)
